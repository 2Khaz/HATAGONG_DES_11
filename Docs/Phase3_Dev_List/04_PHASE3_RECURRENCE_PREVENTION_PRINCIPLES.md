# Phase 3 재발 방지 작업 원칙

## 1. Production 구현 우선

- 사용자에게 필요한 실제 게임 기능을 먼저 구현한다.
- 검증 코드는 Production 구현을 지연시키거나 대체할 수 없다.
- 실제 Scene/EventSystem 연결은 실제 Production 계층을 기준으로 판단한다.

## 2. 단계별 범위 분리

- 한 Stage에는 하나의 명확한 목표만 둔다.
- 모바일, 실제 퍼즐 생성, Session Source, 결과 화면을 한 단계에 섞지 않는다.
- 현재 Stage의 시작 조건이 충족되지 않으면 임의 구현하지 않는다.

## 3. 점검 방식

- 기능별 소형 정적 검사와 최종 Play Mode 확인을 사용한다.
- 새 대형 Harness, 공유 Fixture, 가상 Scene을 만들지 않는다.
- 실제 입력 확인을 가상 Input Device의 대규모 시뮬레이션으로 대체하지 않는다.
- Play Mode 확인 항목은 짧고 사용자가 직접 판정할 수 있게 작성한다.

## 4. 결함 수정 원칙

- 실패 원인을 호출 경로로 확정한다.
- 실패 원인 파일만 최소 수정한다.
- 관련 없는 리팩터링과 구조 교체를 하지 않는다.
- 기존 확인 완료 기능의 계약을 먼저 기록하고 회귀를 막는다.
- 임시 로그가 필요하면 지점과 횟수를 제한하고 사용 후 제거한다.

## 5. 반복 실패 처리

- 동일 원인 또는 동일 구조에서 반복 실패하면 즉시 보수를 중단한다.
- Production 결함, 환경 문제, 검증 방식 문제를 다시 분리한다.
- 구조 재검토 결과와 새로운 최소 수정 범위를 사용자에게 먼저 보고한다.
- 실패 횟수를 줄이기 위한 억지 수치 조정이나 계약 삭제를 하지 않는다.

## 6. 사용자 승인 Gate

- 사용자 승인 전 다음 Stage를 시작하지 않는다.
- 사용자 승인 전 Git add/commit/push를 수행하지 않는다.
- 승인되지 않은 Scene Setup, Package 변경, Branch 변경을 하지 않는다.
- 명시적 미검증 항목을 완료로 보고하지 않는다.

## 7. 완료 보고 원칙

완료 보고에는 반드시 다음을 구분한다.

- 사용자 실제 확인 완료
- 코드 계약 확인
- 미검증
- 후속 단계
- 수정 파일
- 실행하지 않은 작업
- Git 상태
