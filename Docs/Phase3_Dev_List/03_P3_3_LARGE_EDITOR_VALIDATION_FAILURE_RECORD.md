# P3-3 대형 Editor Validation 실패 기록

- 기록일: 2026-07-14 KST
- 목적: 실패 과정을 숨기지 않고 재발 방지 근거로 남긴다.

## 1. 발생한 일

P3-3 작업 중 본래 기능 구현 대신 458 Assertion 대형 Harness가 과도하게 확대됐다.

그 과정에서 실제 Production 게임 흐름과 별개인 다음 요소가 추가됐다.

- 임시 Scene
- Camera
- RenderTexture
- 가상 Input Device
- 공유 Fixture

검증 환경 자체의 Scene 활성화, Graphic 등록 시점, Raycast, 화면 좌표, 입력 장치, Cleanup 문제를 해결하기 위한 보수가 반복됐다.

## 2. 영향

- Harness 반복 보수와 사용자 수동 재실행으로 작업시간과 수면시간 손실이 발생했다.
- 실제 Production 구현이 지연됐다.
- 사용자가 확인해야 할 대상이 게임 기능이 아니라 검증 환경의 동작으로 바뀌었다.
- 실패 수치 감소가 실제 게임 기능 완성을 의미하지 않는 상황이 생겼다.
- 정적 보고와 실제 Play Mode 동작 사이의 불일치가 발생했다.

## 3. 최종 처리

해당 Harness는 최종 제거됐다.

- 삭제된 파일을 복구하지 않는다.
- 유사한 대형 Harness로 대체하지 않는다.
- 가상 Scene, 공유 Fixture, 가상 Input Device를 다시 구성하지 않는다.
- 현재 남아 있는 P3-1/P3-2의 기존 소형 Editor 검사는 기준선으로 보존한다.

## 4. 근본 문제

기능 구현과 검증 환경 구축의 범위가 분리되지 않았다.

하나의 Editor 실행에서 Production UI, Scene 수명, 가상 입력, 렌더링, Cleanup, 계약 수백 개를 모두 증명하려 하면서 검증기 자체가 별도 제품 수준의 복잡도를 가졌다. 그 결과 실패 원인이 Production인지 Harness인지 구분하는 비용이 커졌다.

## 5. 교훈

- 메서드 존재나 검사 성공만으로 실제 UI 입력 연결을 완료로 판정하지 않는다.
- 사용자 Play Mode 결과가 정적 보고와 다르면 사용자 결과를 우선 기준으로 삼는다.
- 실제 실패 지점에서 생성 객체, Raycast 대상, EventSystem callback, 상태 전환을 한 경로로 추적한다.
- 검증 환경을 고치는 대신 Production 결함 파일을 최소 수정한다.
- 반복 실패하면 동일 구조의 보수를 중단하고 검증 방식 자체를 재검토한다.

## 6. 현재 상태

- 대형 Harness: 제거됨
- 복구 계획: 없음
- 대체 대형 Harness 계획: 없음
- P3-3 확인 방식: 기능별 정적 점검 + 사용자 실제 Play Mode 확인
